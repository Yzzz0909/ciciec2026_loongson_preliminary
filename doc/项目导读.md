# 项目导读

这份文档用于帮助第一次接手这个仓库的人快速建立整体认识。

如果把这个项目看成一台正在搭建中的 LoongArch32R 小型 SoC，那么最重要的主线可以先简化成：

`CPU -> Axi_CDC -> AxiCrossbar -> RAM / UART`

先把这条主线看懂，后面再补 `confreg`、中断、DVI、游戏等功能，会轻松很多。

## 1. 这个项目本质上在做什么

这个仓库不是普通软件项目，而是一套龙芯杯初赛的 SoC 实验工程。

它的目标是把下面这些部分拼成一个能运行软件、能做仿真、也能最终上板验证的系统：

- `CPU`：`openLA500`
- `总线互连`：AXI Crossbar
- `主存`：BaseRAM / ExtRAM
- `串口`：UART
- `控制外设`：confreg
- `显示外设`：DVI

项目推进时，建议不要一开始就同时看所有外设。更合理的顺序是：

1. 先跑通 CPU 访问 RAM
2. 再跑通 UART 输出
3. 再补 confreg / 中断
4. 最后再做 DVI 和比赛展示功能

## 2. 建议先按目录理解工程

### `rtl/`

硬件 RTL 主体都在这里，最重要的入口是 [soc_top.v](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/rtl/soc_top.v)。

它相当于整个 SoC 的总装文件，负责：

- 实例化 CPU
- 实例化 AXI CDC
- 实例化 AXI Crossbar
- 实例化 RAM / UART / 其他外设
- 对接板级 IO

### `sim/`

仿真相关文件在这里，最重要的是 [mycpu_tb.v](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/sim/mycpu_tb.v) 和 [sram.v](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/sim/sram.v)。

`mycpu_tb.v` 负责：

- 产生时钟和复位
- 实例化 `soc_top`
- 挂接仿真 SRAM
- 监听 UART 输出
- 判断测试是否结束

### `fpga/`

Vivado 工程脚本和约束都在这里，最关键的是 [create_project.tcl](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/fpga/create_project.tcl) 和 [soc.xdc](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/fpga/constraints/soc.xdc)。

### `sdk/`

软件工具链、BSP、示例程序都在这里。软件最终会被编译成 `axi_ram.mif`，再被仿真 SRAM 读入，所以它和 `sim/` 是一条完整的软件加载链路。

## 3. 最重要的几个文件分别干什么

### `rtl/soc_top.v`

这是当前最关键的文件。它最值得你重点理解的地方有 4 个：

1. CPU AXI 主设备接口是怎么接出来的
2. `Axi_CDC` 怎样把请求跨时钟域送到系统总线侧
3. `AxiCrossbar_2x8` 怎样按地址把请求分发到不同从设备
4. RAM / UART / confreg / DVI 这些从设备最终怎么挂上去

### `rtl/ip/open-la500/`

这是 CPU 核目录。建议优先看 [mycpu_top.v](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/rtl/ip/open-la500/mycpu_top.v) 以及五级流水的各 stage 文件。

### `sim/mycpu_tb.v`

这个文件非常重要，因为它不只是仿真环境，还会隐含一些结构要求。比如之前报错中的 `u_soc_top.u_axi_uart_controller`，就说明 testbench 会通过层级路径访问 `soc_top` 内部实例。

## 4. 现在这版工程最适合怎么读

建议把 `soc_top.v` 拆成 5 层来看：

- 顶层 IO 层：时钟、复位、SRAM、UART、LED、数码管、视频输出
- CPU 主设备层：CPU 发起 AXI 请求的地方
- 跨时钟层 `Axi_CDC`
- 地址分发层 `AxiCrossbar_2x8`
- 从设备层：现阶段最关键的是 `axi_wrap_ram_sp_external` 和 `axi_uart_controller`

## 5. 一个程序是怎样跑起来的

以 `hello_world` 为例，流程大致如下：

1. 在 `sdk/software/examples/hello_world` 下执行 `make`
2. 生成 `hello_world.elf`、`hello_world.bin`
3. 转换得到 `sdk/axi_ram.mif`
4. 仿真时 [sram.v](/C:/Users/naozh/Desktop/Task/3-2/Loogarch/ciciec2026_loongson_preliminary/sim/sram.v) 通过 `$readmemb` 读入镜像
5. CPU 从 RAM 取指执行
6. 程序访问 UART 寄存器
7. testbench 把 UART 输出打印到控制台

## 6. 现阶段哪些模块必须先跑通

当前主线必须优先完成：

- `core_top`
- `Axi_CDC`
- `AxiCrossbar_2x8`
- RAM 包装模块
- UART 控制器

目前可以先占位或后补：

- `confreg`
- 中断控制器
- DVI
- 游戏相关显示逻辑
