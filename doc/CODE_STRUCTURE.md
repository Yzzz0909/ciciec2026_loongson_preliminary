# Code Structure Guide

这份文档的目标不是重复 `README.md` 的总览，而是帮助你从“代码怎么组织”的角度理解这个项目，尤其是 `rtl/soc_top.v` 这层顶层 SoC 连接。

## 1. 先抓主线：这个项目本质上在做什么

这个仓库当前最核心的事情，是把一个 LoongArch CPU 核接进 SoC 顶层，让它能够：

1. 从 RAM 取指、读写数据
2. 通过 AXI 总线访问外设
3. 在仿真里先跑通 UART 打印
4. 后续再逐步补比赛要求的控制外设、显示外设、板级接口

所以读代码时，不要一开始就被所有外设、所有 IO、所有 AXI 信号吓住。真正要先盯住的是这条最小可运行链路：

`CPU -> AXI CDC -> AXI Crossbar -> RAM / UART`

只要这条链路跑通，项目就从“静态代码”变成“能执行程序的 SoC”。

## 2. 顶层文件分工

项目里建议先按下面这个层次理解：

### `rtl/ip/open-la500/mycpu_top.v`

这里是 CPU 核心实现，当前工程里实际实例化使用的是 `core_top`。

你可以把它理解为：

- 负责执行指令
- 发出 AXI 读写请求
- 输出调试信息，比如 `debug_wb_pc`

这部分一般不是现在最先要大改的地方。前期我们的重点是让它“能通过 SoC 正常访问外部世界”。

### `rtl/soc_top.v`

这是当前最重要的文件，相当于整个系统的“总装图”。

它做的事情主要有：

- 实例化 CPU
- 实例化 AXI 跨时钟模块 `Axi_CDC`
- 实例化 AXI 互连 / 交叉开关 `AxiCrossbar_2x8`
- 实例化 RAM 包装模块
- 实例化 UART 控制器
- 对接板级 IO，比如 SRAM、串口、LED、数码管、视频输出等

如果以后你要判断“某个功能应该写在哪里”，通常先问自己：

- 是 CPU 内部行为？那看 `mycpu_top` / `core_top`
- 是 SoC 级模块连接、地址分配、外设接线？那就在 `soc_top.v`
- 是某个独立外设实现？那去对应的 `rtl/ip/...` 或外设模块文件

### `sim/mycpu_tb.v`

这是仿真 testbench。

它的作用不是实现功能，而是：

- 产生时钟、复位
- 给 `soc_top` 提供仿真环境
- 监听 UART 输出
- 判断测试是否结束

之前 Vivado 报错里提到的 `u_soc_top.u_axi_uart_controller`，就是 testbench 在按层级路径访问 `soc_top` 内部的 UART 实例。

这也说明一个很关键的点：

仿真 testbench 有时会“依赖实例名”。

所以顶层里某些模块不仅要连对，实例名也要和 testbench 预期一致。

## 3. 现在这版 SoC 应该怎么看

你可以把 `soc_top.v` 拆成 5 个逻辑区域来看。

### A. 外部接口层

最上面是顶层端口，也就是 FPGA 板子或仿真环境能看到的信号，比如：

- 时钟、复位
- BaseRAM / ExtRAM 信号
- UART 串口
- LED、数码管
- 视频输出

这一层的意义是“和外部世界连接”，不是系统内部逻辑本体。

### B. CPU AXI 主设备层

CPU 跑起来后，会通过 AXI 发请求。这里常见的就是：

- `ar*` / `r*`：读地址、读数据通道
- `aw*` / `w*` / `b*`：写地址、写数据、写响应通道

你后面在看波形时，第一批最重要的信号其实就是这里这一组。

### C. 跨时钟层 `Axi_CDC`

这个模块的作用是把 CPU 侧 AXI 请求，从一个时钟域安全地送到系统总线侧。

所以它不是“功能外设”，而是“通路模块”。

判断它有没有工作，最简单的方法就是看：

- CPU 侧发出的 `cpu_*`
- CDC 输出后的 `cpu_sync_*`

如果前面有请求、后面没有，问题大概率就在 CDC 或 CDC 周边连接。

### D. AXI 互连层 `AxiCrossbar_2x8`

这个模块负责地址译码和主从设备分发。

它做的事情可以理解成：

- CPU 发来一个 AXI 请求
- Crossbar 看地址属于哪一段
- 把请求送去 RAM、UART 或其他外设

所以它决定的是“这个地址最终去找谁”。

### E. 从设备层

也就是被访问的设备，现阶段最关键的是两个：

#### RAM

`axi_wrap_ram_sp_external`

作用：

- 把 AXI 访问翻译成外部 SRAM 风格接口
- 给 CPU 提供指令和数据存储空间

这是最基础的设备，因为程序能不能继续跑，首先取决于 RAM 读写通不通。

#### UART

`axi_uart_controller`

作用：

- 把 AXI 写操作转成串口寄存器访问
- 在仿真中常用来打印字符串

当前阶段如果一切正确，最常见的里程碑就是看到串口输出，比如 `Hello Loongarch32r!`

## 4. 现在哪些模块是“先占位，后补全”

根据目前的推进方式，项目不是一次性把所有模块都完成，而是分阶段补。

现阶段可以把模块分成两类：

### 已经属于主线必须跑通的

- CPU `core_top`
- `Axi_CDC`
- `AxiCrossbar_2x8`
- RAM 包装模块
- UART 控制器

### 目前可能先 tie-off 或占位的

- `confreg`
- LED / 数码管相关逻辑
- DVI / 视频输出
- 更完整的板级交互外设

这些不是不重要，而是当前优先级低于“先让 CPU 真正跑起来”。

## 5. 按什么顺序写代码最合理

如果我们从工程推进角度来排顺序，建议严格按照“先主通路，后扩功能”的顺序。

### 阶段 1：跑通最小执行链路

目标：

- CPU 能从 RAM 取指
- AXI 请求能穿过 CDC
- Crossbar 能把地址送到正确从设备
- UART 能被访问

验证标准：

- 仿真不报 elaboration 错误
- 波形里读请求能到 RAM
- 能看到 UART 写
- 最好能看到串口打印

### 阶段 2：补控制类外设

目标：

- 加入 `confreg` 一类配置 / 控制寄存器
- 让程序能读开关、写 LED、驱动数码管

验证标准：

- 地址访问正确
- 寄存器读写行为正确
- 板级 IO 或仿真观测值正确变化

### 阶段 3：补显示或更复杂外设

目标：

- 加入视频 / DVI / 图形输出相关链路
- 处理更复杂的时序与带宽问题

验证标准：

- 时钟域关系清晰
- 输出时序满足要求
- 功能演示可见

### 阶段 4：软硬件联调

目标：

- 软件程序按 SoC 地址映射正确访问硬件
- 工具链、链接脚本、程序镜像、初始化流程匹配

验证标准：

- 程序行为与硬件设计一致
- 可以稳定复现实验 / 比赛要求的展示结果

## 6. 你现在最该盯住的几个关键点

如果你还没有完全看懂项目，不要试图一次看完所有 RTL。先盯住下面几个“抓手”。

### 抓手 1：一条读请求到底怎么走

看这一条链：

`core_top -> Axi_CDC -> AxiCrossbar_2x8 -> RAM -> 返回 rdata`

如果这条读链理解了，整个 SoC 的主骨架就清楚一半了。

### 抓手 2：地址是谁在决定去向

重点看 Crossbar 和地址映射。

因为 SoC 里很多“功能问题”，最后都不是模块没写，而是地址没分对、连错设备、响应没回来。

### 抓手 3：testbench 到底在观察什么

`sim/mycpu_tb.v` 其实在告诉你：

- 它关心 UART 输出
- 它关心测试是否结束
- 它可能依赖顶层内部实例名

所以 testbench 不只是“运行环境”，它本身也是项目约束的一部分。

## 7. 现阶段建议的阅读顺序

如果你准备开始真正上手代码，建议按这个顺序看：

1. `README.md`
2. `PROJECT_STAGES.md`
3. `rtl/soc_top.v`
4. `sim/mycpu_tb.v`
5. `rtl/ip/open-la500/mycpu_top.v`
6. `rtl/ip/Bus_interconnects/Axi_CDC.v`
7. RAM / UART 对应模块

这样看会比较顺，因为你是先知道“系统要干什么”，再看“顶层怎么接”，最后再深入具体模块。

## 8. 现阶段最实际的开发策略

不要一上来追求“所有外设都接完、所有功能都能跑”。

更稳妥的方式是：

1. 先确保最小 SoC 数据通路稳定
2. 再补地址映射
3. 再补控制外设
4. 最后做显示、上板、性能和完整演示

这会让调试成本低很多。

因为一旦最小链路不稳，后面所有外设问题都会混在一起，很难定位。

## 9. 一句话总结

这个项目当前最重要的不是“把所有模块都看懂”，而是先建立这样一个清晰认识：

`soc_top.v` 是总装图，CPU 是主设备，AXI 是通路，Crossbar 是分发中心，RAM/UART 是现阶段最关键的第一个落地点。

只要你能顺着这条主线去读代码，整个项目就会越来越清楚。
